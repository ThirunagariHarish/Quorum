# Release Notes тАФ v0.3.0

**Date:** 2026-04-19
**Type:** Minor release тАФ mega regression + security fix release
**Versions:** `frontend@0.3.0` ┬╖ `API v1.2.0`

---

## Highlights

### 1 ┬╖ Critical Security Hardening (P0)
Two P0 security issues are resolved in this release. `JWT_SECRET` and `ENCRYPTION_KEY` no
longer have insecure hardcoded defaults тАФ the application now refuses to start if either
variable is missing from the environment. Additionally, `ENCRYPTION_KEY` now enforces a
strict 32-byte length requirement instead of silently truncating or padding the value,
preventing a class of subtle encryption bugs.

### 2 ┬╖ IDOR & SQL Injection Fixes
Six IDOR (Insecure Direct Object Reference) vulnerabilities in `reviews.py` are patched тАФ
every endpoint now JOINs on Paper and filters by the authenticated `user_id`. The `sort_by`
SQL injection surface in `papers.py` is replaced with an `ALLOWED_SORT_COLS` allowlist.

### 3 ┬╖ WebSocket Reliability & Frontend API Client
The frontend WebSocket client gains a proper `onerror` handler and exponential backoff
reconnect logic (1 s тЖТ 16 s, max 5 attempts), with a ghost-connection leak fixed via an
`alive` flag on unmount. The `compilePaper()` function is fully rewritten inside `ApiClient`
with the correct JSON field names and Bearer token injection тАФ it was previously broken in
three independent ways.

### 4 ┬╖ Async I/O & Agent Correctness
All five MinIO methods in `storage.py` are wrapped with `asyncio.to_thread`, unblocking the
event loop. Paper generation agents (`spawn_scouts`, `spawn_researchers`) are now
parallelised with `asyncio.gather`, and agent task assignment is made fully atomic with
`.with_for_update()`. `IEEE_SYSTEM_PROMPT` тАФ previously imported but never forwarded тАФ is
now passed to all four LLM calls in `ieee/agent.py`.

## Breaking Changes

### тЪа Required environment variables (deployment action needed)

`JWT_SECRET` and `ENCRYPTION_KEY` **must** now be set explicitly in your `.env` /
`.env.production` file. The application will fail to start if either is absent.
See **Migration Steps** below.

---

## Migration Steps

### 1 ┬╖ Set required secrets (mandatory before deploying)

Add the following to your production `.env` / secret store if not already present:

```bash
# Generate a strong JWT secret (тЙе 32 random bytes, base64 or hex)
JWT_SECRET=$(openssl rand -hex 32)

# Generate exactly 32 bytes for the encryption key
ENCRYPTION_KEY=$(openssl rand -hex 16)   # 16 bytes hex = 32 ASCII chars
```

> **Important:** `ENCRYPTION_KEY` must be **exactly 32 bytes** (characters) long.
> Values that are shorter or longer will cause the application to refuse to start.

### 2 ┬╖ Frontend dependency install (if running outside Docker)

No new frontend packages were added in this release. An `npm install` is not required unless
your `node_modules` is stale.

### 3 ┬╖ No database schema changes

No Alembic migrations are included. The database upgrade step can be skipped.

## Rollback Plan

### Rollback procedure (Docker Compose)

If this release introduces a regression in production, roll back to `v0.2.0` by:

```bash
# On the VPS / deployment host
cd /opt/quorum

# 1. Identify the v0.2.0 tag SHA or the pre-release HEAD SHA
git fetch origin --tags
git log --oneline v0.2.0   # confirm target SHA

# 2. Reset to the prior release
git checkout v0.2.0

# 3. Rebuild and restart the affected services
docker compose -f docker-compose.prod.yml build --no-cache backend frontend
docker compose -f docker-compose.prod.yml up -d --no-deps backend frontend

# 4. Confirm health
curl -sf http://localhost:80/api/v1/health | python3 -m json.tool
```

**Pre-release HEAD SHA:** `49b7fa0` (`fix(frontend): resolve all Cortex BLOCKER/MUST-FIX items from review`)

> **Note on secrets:** Rolling back to v0.2.0 re-enables the old hardcoded defaults for
> `JWT_SECRET` and `ENCRYPTION_KEY`. If you have already rotated these secrets in
> production, you **must** keep the new values in your environment; do **not** rely on the
> old defaults after a rollback.

Infrastructure services (postgres, redis, minio) are unaffected тАФ no schema changes were
made in this release, so no data rollback is needed.

### Rollback verification

After rolling back, confirm the API version reports `1.1.0`:

```bash

# expected: 1.1.0
```
